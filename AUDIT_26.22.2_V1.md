## 🐞 Исправлено в 26.22.2 V1

### 🔴 «Умная античиталка» — 3 бага в 26.22.2 стоке
На новом стоке (build 6779) при `smartAntiRead=ON` возникали три симптома:
- **Bug A** — отметка «прочитано» уходила собеседнику **сразу** при открытии чата, а не после ответа.
- **Bug B** — при повторном открытии чата все входящие сообщения отображались непрочитанными (счётчик и «N новых»).
- **Bug C** — при открытии чата список не прокручивался в конец (оставался сверху / посередине).

**Корневые причины (все три — переезд chokepoint'ов в 26.22.2):**
1. **Bug A** — в стоке появился **второй путь** отправки read-mark по WS: батч-flush в `pte.C` (ReadMarkSender). Раньше read-marks шли через `wjd.c`, теперь бо́льшая часть — через batch. Прежний гейт `wjd.c` их не ловил.
2. **Bug B** — метод `wjd.a` (`markChatAsRead`), в котором стоял `markChatOpened`, на пользовательском open чата **больше не вызывается** — реальный driver теперь `c3/qza.smali::a(Loza;Lp55;)V` (2 параллельных сайта вызова `Lwjd;->d(...)V`). Инжектор `markChatOpened` шёл в мёртвую ветку, `openedChats:HashSet<Long>` не наполнялся.
3. **Bug C** — event-holder события scroll переехал `Llge;` → `Lege;`, семантика `v1` в `Lyca;->c()` сменилась (`v1 = list.size()`, не `size-1`). Прежний якорь промахивался.

**Фикс:**
- **`apply_v_smart_antiread_second_path.py`** — GATE-2 на `Lpte;->C` перед batch-flush (chokepoint истинного WS-send).
- **`apply_v_smart_antiread_qza_open.py`** — `markChatOpened(J)V` в реальном chat-open driver `qza.a()` (оба сайта, sentinel `:qza_mark_chat_opened_a` и `_b`).
- **`apply_v_smart_antiread_chat_open.py`** — резервный вызов в `wjd.a` (на случай если стоковый flow вернёт вызов).
- **`apply_v_smart_antiread_scroll.py`** — обновлён под `Lege;->g:J == -1` initial-open sentinel в 26.22.2.

### 🔄 Расширение антидетект-логики на обычный `antiRead`
Внутричатовое обнуление счётчика непрочитанных и scroll-to-bottom при открытии теперь работают **не только для `smartAntiRead`, но и для обычного `antiRead`**. Раньше при `antiRead=ON` (без smart) сохранялся визуальный баннер «N новых» в открытом чате и не работал автоскролл. Причина: гейт `unreadGateActive(J)Z` в `AppFlags` проверял только `smartAntiRead:Z`. Добавлена OR-ветка на `antiRead:Z`.

### 📁 `startOnNew` — RE-ENABLED
Тумблер «Стартовая папка Новые» был **DEFER в 26.22.0 и 26.22.1** (body-drift после bump'а). В 26.22.2 переязван на `gg3.pswitch_2b6` (after `Lxr6;->M`); restore-guard (`gateRestoreFolder`) вернулся — при перезапуске приложения активная папка не перетирается на «Все».

### 🔎 «Проверить обновления» — ручная проверка новой версии
Short-tap по строке версии в **Настройки → О приложении → «Версия ...»** открывает диалог с текущей и последней доступной версией и кнопкой перехода на страницу релиза. Проверка идёт через **отдельный mirror-репозиторий** (уменьшенная копия), URL зашит в APK. Long-tap по версии по-прежнему открывает дев-меню.

> **Автопроверка обновлений НЕ включена.** Никаких фоновых запросов при старте приложения — только по явному тапу.

### 🇷🇺 Госуслуги баннер — реальный чокпоинт
В 26.22.1 гейт на `hideDigitalId` стоял на первом `addAll` в `c3/pye.smali:1685`, но hero-карточка «госуслуги» (лого 98dp×18dp) в шапке Settings-таба **не блокировалась**, потому что она рендерится **другим кодом** — server-driven цикл по `PmsProperty Lkic;->b2:Liic;` index `0x9d`, идущий сразу после первого блока.

**Фикс**: добавлен **второй патч** `apply_v_hide_gu_banner_v2.py` — гейт на реальный чокпоинт (метка `:goto_5c3`, before `invoke-interface {v12}, Lth8;->getValue()`), прыжок на штатный empty-list branch `:cond_5e6`. Оба патча под одним тумблером `hideDigitalId`.

> **Известное ограничение**: PmsProperty `0x9d` управляет одновременно и hero-баннером, и пунктом «Цифровой ID» в списке настроек. Разделить их на два тумблера в 26.22.2 не удалось (общий серверный ключ) — один тумблер `hideDigitalId` управляет **обоими элементами**.

## 🛡 Профиль приватности

### Невидимка (`invisible`) — мастер-переключатель
Взводит дочерние приватные флаги; каждый работает и отдельно.

| Дочерний флаг | Что делает |
|---|---|
| Антипечатанье (`antiTyping`) | собеседник не видит набор текста |
| Античиталка (`antiRead`) | собеседник не видит отметку «прочитано» |
| Умная античиталка (`smartAntiRead`) | «прочитано» уходит только ПОСЛЕ вашего ответа |
| Антиприсутствие (`antiPresence`) | онлайн / last-seen скрыты (сервер видит offline) |

**Взаимоисключение**: `antiRead` и `smartAntiRead` не могут быть включены одновременно — включение одного снимает другой. **26.22.2 V1**: и `antiRead`, и `smartAntiRead` теперь одинаково блокируют внутричатовый счётчик «N новых» и скроллят в конец при открытии.

### Анти-удаление (`antiDelete`) — 4 слоя + офлайн-персист

1. **L3 — Central WS chat-event**: live-канал, собеседник удаляет при онлайн-приложении.
2. **L4 — Room ORM body-clear**: локальный SQL-сброс тела блокируется.
3. **V3 persist**: удалённые сообщения остаются видимыми после ре-синка.
4. **Маркер «· удалено»** в строке времени + **время удаления** (`dk_times`).
5. **Офлайн-удаления через FCM** — если удаление пришло, пока приложение убито OEM-killer'ом (RealMe/ColorOS/MIUI: «Очистить всё», «Force stop», hard-kill), метка восстанавливается при следующем старте.

Общий гейт `antiDelete`: **ON** → защита; **OFF** → поведение стока.

## ⚠ Честно: чего НЕТ в 26.22.2 V1

- **Тумблер `improveNotifications` временно недоступен** (скрыт из UI). Причина крайних крашей — жёсткие проверки Android 15 / OxygenOS 15 на IMPORTANCE фонового канала. Вернём в 26.23.x после device-matrix smoke.
- **`hideFavorites` временно DEFER** — рефактор chat-list combine-оператора в 26.22.2, тумблер в UI **есть**, но эффекта не даёт до следующей версии.
- **2 транспортные нейтрализации остаются DEFERRED** до 26.23.x:
  - `WS-redirect` — используется штатно для sharding SMS-регистрации; безусловный skip ломает вход по новым номерам.
  - `vmk` (SSL-session-validate / DNS-client-config) — реестр `PmsProperty` изменил тип default'ов, тривиальная замена вызывает `ClassCast`.
- **Разделение тумблеров «Цифровой ID» и «Госуслуги баннер»** — оба контролируются одним `hideDigitalId` (общий серверный ключ PmsProperty `0x9d`).
- **Требуется one-time `pm clear ru.oneme.ap2`** после установки APK клона (штатный шаг установки клона).
- **Минимальная версия Android — 8.0** (API 26). На Android 5.0.1 / 6.0 / 7.x установка сорвётся с ошибкой «Сбой при разборе пакета» — это ограничение стока, не мода.
