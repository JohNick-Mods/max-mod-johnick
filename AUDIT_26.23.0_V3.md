## 🆕 Новое в 26.23.0 V3

### ⌚ Watch-mode — авто-адаптация под смарт-часы (реализовано и работает)

**Проблема**: стоковые layouts MAX рассчитаны на телефоны (ширина ≥ 360dp). На устройствах с маленьким экраном (Huawei Watch 3/4 Pro, WearOS-часы: 1.43" ≈ 466×466 px) элементы интерфейса вылезают за края круглого/квадратного экрана — использовать мессенджер невозможно.

**Решение**: в `OneMeApplication.attachBaseContext(Context)` вставлен inline-гейт, который при старте приложения:

1. Читает `Resources.getDisplayMetrics().widthPixels` устройства.
2. Если ширина **< 500 px** — создаёт override-конфигурацию через `Context.createConfigurationContext(Configuration)` с:
   - `densityDpi = 240` (hdpi)
   - `smallestScreenWidthDp = 310`
   - `screenWidthDp = 310`
3. Возвращает override-контекст в `super.attachBaseContext(...)`. Дальнейший layout inflation идёт уже с ужатой плотностью → 466 px / 240 dpi × 160 ≈ 310 dp → интерфейс умещается на маленьком экране.
4. На телефонах и планшетах (ширина ≥ 500 px) гейт не срабатывает — контекст передаётся в super без изменений.

**Ключевое:** пересоздание контекста идёт **на уже локализованном контексте** (после `LocaleHelper`), а не на исходном — поэтому язык интерфейса при переключении плотности **сохраняется**.

**Без тумблера.** Активация полностью автоматическая по разрешению экрана — на телефонах эффекта нет, на носимых устройствах включается сам. Это исключает риск случайного включения на смартфоне и не требует дополнительного управляющего флага.

**История:** watch-mode был анонсирован в 26.23.0 V1, но из-за ошибки сборки (регистры v20-v26 в 4-битных smali-инструкциях 22c/22t/35c — за пределами лимита v0-v15) блок отвергался ассемблером и **фактически не попадал в собранный APK**. На телефонах баг был незаметен. В V2 неработающий код был убран из пайплайна. В **V3** реализация переписана с валидным регистр-планом (`v4/v5/v8/v9/v14/v15`) — проверена в дизассемблированном классе `.dex` и на устройстве smoke-тестом (0 VerifyError на холодном старте).

## 🛡 Профиль приватности

### Невидимка (`invisible`) — мастер-переключатель
Взводит дочерние приватные флаги; каждый работает и отдельно.

| Дочерний флаг | Что делает |
|---|---|
| Антипечатанье (`antiTyping`) | собеседник не видит набор текста |
| Античиталка (`antiRead`) | собеседник не видит отметку «прочитано» |
| Умная античиталка (`smartAntiRead`) | «прочитано» уходит только ПОСЛЕ вашего ответа |
| Антиприсутствие (`antiPresence`) | онлайн / last-seen скрыты (сервер видит offline) — независимый тумблер |

**Взаимоисключение**: `antiRead` и `smartAntiRead` не могут быть включены одновременно. Оба блокируют внутричатовый счётчик «N новых» и скроллят в конец при открытии.

### Анти-удаление (`antiDelete`) — 4 слоя + офлайн-персист

1. **L3 — Central WS chat-event**: live-канал, собеседник удаляет при онлайн-приложении.
2. **L4 — Room ORM body-clear**: локальный SQL-сброс тела блокируется.
3. **V3 persist**: удалённые сообщения остаются видимыми после ре-синка.
4. **Маркер «· удалено»** в строке времени + **время удаления** (`dk_times`).
5. **Офлайн-удаления через FCM** — метка восстанавливается при следующем старте, если удаление пришло при убитом OEM-killer'ом приложении.

Восстановленные удалённые сохраняются с `isRead=1` — не задирают счётчик непрочитанных. Общий гейт `antiDelete`: **ON** → защита; **OFF** → поведение стока.

### 📢 Каналы = сток при включённых античиталках
При `antiRead`/`smartAntiRead`/`invisible` каналы ведут себя как в стоке: read-mark для каналов всегда уходит на сервер.

### 🔕 Приваси-стабы (P0/P1)
- **myTracker (VK)** — вызовы SDK нейтрализованы (`return-void`).
- **Фоновая выгрузка контактов** — метод нейтрализован (`return-void`).
- **NFC-HCE** — APDU-ответ `SW 6F00` (сервис не обслуживается).
- **TrustManager accept-all** — доверие «любому» сертификату заменено на `throw`.
- **No-IP телеметрия** — IP-адрес не отправляется в телеметрию соединения.
