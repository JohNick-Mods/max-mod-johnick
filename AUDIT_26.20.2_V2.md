## 1.2 Новые и доработанные функции

- **Новый 15-й переключатель — «Скрыть номер телефона от оператора» (`blockMobileIdVerify`).** В 26.18.1 было 14 переключателей. Новый блокирует скрытое определение номера через оператора (Header Enrichment / Mobile ID) — запрос MSISDN, уходящий по сотовому каналу. По умолчанию **выключен**. На «Цифровой ID» (Госуслуги) не влияет — это разные механизмы (см. 1.4-a).
- **«При открытии — папка "Новые"» (`startOnNew`) доведён до полноценной работы.** В 26.18.1 функция была частичной. Теперь реализована через observer-хук списка папок (`c1/qe3` :pswitch_2b5 → `selectStartFolder`) и работает как в клоне, так и в основном приложении (см. 1.4-b).

## 1.3 Изменения в безопасности относительно 26.18.1

- **NSC: операторские HE/Mobile-ID домены заблокированы, гос-эндпоинты оставлены доступными.** В стоке 26.20.2 разработчик расширил привязку номера к гос-системам — добавил cleartext-домены `nuc-cdp.voskhod.ru` (НИИ «Восход») и `nuc-cdp.digital.gov.ru` (Минцифры / Цифровой профиль). Операторские домены Header Enrichment / Mobile ID (Мегафон, МТС, Tele2, Билайн) мод переводит в `cleartextTrafficPermitted="false"` — скрытый MSISDN-запрос по HTTP падает. **Гос-эндпоинты намеренно оставлены cleartext-доступными** — они задействованы штатной верификацией «Цифрового ID»/Госуслуг, и их блокировка ломала привязку (см. 1.4-a). Управление скрытым запросом номера вынесено на переключатель `blockMobileIdVerify`.
- **FCM-фильтр стал управляемым гейтом (`gateRemoved`).** Типы `MessageRemoved`/`ChatMessageRemoved` теперь честно соблюдаются при выключенной антиудалялке (удаление «у всех» работает как в стоке) и блокируются только при включённой — вместо безусловного NOP. Опасные `LocationRequest` (silent-GPS) и `TamtamSpam` блокируются безусловно.
- **Изоляция device_id клона.** Клон (`ru.oneme.ap2`) получает собственный device_id (пустая строка в `DeviceIdRemoteDataSource` → клиент генерирует свой UUID), что чинит «удалить у всех» с клона и доставку push на клон при общей подписи с основным.

## 1.4 Исправления (2026-06-28)

**(a) Восстановлена работа «Цифрового ID» (Госуслуги/ЕСИА).**
Симптом: при создании Цифрового ID после авторизации в Госуслугах вместо возврата в MAX открывался браузер с ошибкой **«Не нашли чат по этой ссылке»**. Уже созданный ID при обновлении 26.18.1 → 26.20.2 сохранялся (хранится в данных приложения) — ломалось именно создание/возврат callback.

Корневая причина — **побочный эффект приватностного патча P0-2 (myTracker)**. Этот патч нейтрализует трекер VK, обнуляя его инициализацию (`MyTracker.initTracker → return-void`). Но MAX использует другой метод того же класса — `MyTracker.handleDeeplink(Intent)` — как **первый шаг разбора любой входящей ссылки** (`LinkInterceptorActivity`). При неинициализированном трекере `handleDeeplink` возвращает `null` («MyTracker hasn't been initialized yet»), и обработчик ссылок уходил в запасной путь на главный экран **без URL** — отсюда «не нашли чат». В 26.18.1 патча P0-2 не было (он добавлен начиная с 26.20.1), поэтому там Цифровой ID работал.

Исправление: `MyTracker.handleDeeplink` заменён на безопасный passthrough — возвращает URL напрямую из Intent (`intent.getDataString()`), **без инициализации трекера и без обращений в сеть** (приватность не нарушена, телеметрия по-прежнему мертва). Это чинит **все** внешние ссылки `max.ru` (приглашения, чаты по ссылке, callback Госуслуг), а не только Цифровой ID. Точка: `com/my/tracker/MyTracker` (sentinel `# :p0_2_mytracker_deeplink_passthrough`).

Два сопутствующих условия, без которых callback Госуслуг не доходит до приложения:
- **Разрешение ссылок `max.ru`.** Мод переподписан собственным ключом (иначе его нельзя установить), поэтому Android по умолчанию не доверяет автоматическому открытию ссылок `max.ru` в приложении, и ответ Госуслуг (`https://max.ru/...`) уходит в браузер. Это общее следствие переподписи — касается любого модифицированного MAX. Лечится разовым разрешением (Настройки → Приложения → MAX → «Открывать по умолчанию» → «Открывать поддерживаемые ссылки»). Сборка показывает разовую подсказку с кнопкой перехода в эти настройки (`AppLinkPrompt` на старте). Подробности — в `DIGITAL_ID_README.md`.
- **Cleartext для гос-эндпоинтов** (см. 1.3) — оставлен доступным, иначе шаг верификации обрывается.

> Уточнение: переключатель `blockMobileIdVerify` (патч D2) к этой проблеме отношения **не имеет** — он управляет скрытым MSISDN-запросом через оператора, а не механизмом Госуслуг (подтверждено: переключение тумблера симптом не меняло).

**(b) «При открытии — папка "Новые"» теперь работает и в основном приложении.**
Симптом: папка «Новые» открывалась в свежем клоне, но не в активно используемом основном приложении. Причина — тайминг: приложение восстанавливает последнюю открытую папку (`ChatsTabWidget.onViewCreated` → `getArgs().getString("folder_id")` → выбор по id), и это восстановление выполнялось **после** нашего выбора «Новые», перетирая его. В клоне сохранённой папки нет (`folder_id = null` → ранний выход), поэтому выбор выживал. Исправление: восстанавливаемый `folder_id` пропускается через гейт `AppFlags.gateRestoreFolder()` — при включённом `startOnNew` он возвращает `null`, и штатное восстановление само делает ранний выход, не мешая выбору «Новые». Путь deep-link (`onUpdateArgs`) не затронут. Точка: `ChatsTabWidget.onViewCreated` (sentinel `# :son_restore_gate`).
